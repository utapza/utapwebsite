# iOS Pods folder and Expo autolinking

## Should you remove `ios/Pods` from .gitignore?

**No. Keep `ios/Pods` in .gitignore.**

`ios/Pods` is the iOS equivalent of `node_modules`:

- It holds CocoaPods dependencies (downloaded and built).
- It contains **generated** files, including `Pods/Target Support Files/Pods-uTap/ExpoModulesProvider.swift`.
- It’s large and environment-specific, so it’s not committed. Every machine (and EAS) runs `pod install` to create it.

So: **do not remove `/Pods/` from `ios/.gitignore`.**

---

## How does `ExpoModulesProvider.swift` get recreated?

It’s **regenerated every time** someone runs `pod install` (or the equivalent step in EAS).

Rough flow:

1. **Podfile** uses `use_expo_modules!` and calls into Expo’s autolinking.
2. When **pod install** runs, it runs the Expo autolinking script (e.g. `expo-modules-autolinking resolve --platform ios`).
3. That script scans `node_modules` for Expo modules (using `expo-module.config.json`, podspec, etc.).
4. A **build phase or script** in the Xcode project writes `ExpoModulesProvider.swift` under  
   `ios/Pods/Target Support Files/Pods-uTap/` using that autolinking result.

So:

- **Locally:** After `npm install` and `cd ios && pod install` (or `npx expo prebuild --platform ios`), `ExpoModulesProvider.swift` is generated from your current `node_modules`. No need to commit it.
- **EAS:** The build runs the same pipeline (install deps → prebuild/pod install → Xcode build). So EAS **does** run the step that creates Pods and generates `ExpoModulesProvider.swift`; your Xcode build log only shows the *compilation* of that already-generated file.

The manual edit we did to add `ExpoMifareScannerModule` was a one-off for your local Pods. After the next `pod install` (or prebuild), that file will be **overwritten** by the generator. For the generator to include your module automatically, the **package** must be correct:

- `expo-module.config.json` has `"apple": { "modules": ["ExpoMifareScannerModule"] }` (and we added that).
- The published npm tarball includes `ios/`, the podspec, and the Swift module.

---

## Does EAS run pod install?

**Yes.** EAS Build runs the full native iOS pipeline. That includes:

- Installing JS dependencies (e.g. `npm ci` / `yarn`).
- Prebuild and/or CocoaPods: the step that creates `ios/Pods` and runs the Expo autolinking that generates `ExpoModulesProvider.swift`.
- Then the Xcode build (what you see in the Xcode build log).

The log at `ios/xcode-build-53ed5a6b-380b-4a8a-a41e-51ae23c54585.log` is the **Xcode** build; the “pod install” (or prebuild) step runs **before** that, so it won’t appear in that specific log. The fact that the log references  
`/Users/expo/workingdir/build/ios/Pods/Target Support Files/Pods-uTap/ExpoModulesProvider.swift`  
confirms that on EAS the Pods folder and `ExpoModulesProvider.swift` are created during the build.

## Summary

| Question | Answer |
|----------|--------|
| Remove Pods from .gitignore? | **No.** Keep it ignored, like `node_modules`. |
| How is ExpoModulesProvider recreated? | Automatically every time `pod install` (or prebuild) runs, from autolinking + your packages’ `expo-module.config.json`. |
| Does EAS run pod install? | **Yes.** Before the Xcode build; the Xcode log only shows compilation of the already-generated files. |

To have your module included in the generated provider on EAS and locally, keep the module’s `expo-module.config.json` and podspec correct and **publish** a new version; then reinstall that version in the app and run a new build / `pod install`.

## Why you don't see "ExpoMifareScanner" in OTHER_LDFLAGS (Implicit dependency)

The Xcode build log lists **Implicit dependency on target 'X' in project 'Pods' via options '-lX' in build setting 'OTHER_LDFLAGS'** for every pod that is **linked into the app**. Those entries come from the `Pods-uTap` target's `OTHER_LDFLAGS` in `Pods/Target Support Files/Pods-uTap/Pods-uTap.debug.xcconfig` (and the release variant). That xcconfig is **generated by CocoaPods** when you run `pod install`, using the list of pods that the Podfile (and Expo autolinking) decided to include.

So:

- You see **ExpoModulesCore** (and ExpoSecureStore, ExpoLinearGradient, etc.) because those pods were in the autolinking result and were added to the Podfile's native modules, so they get `-lExpoModulesCore` etc. in OTHER_LDFLAGS.
- You **don't** see **ExpoMifareScanner** because when that build's `pod install` ran, the ExpoMifareScanner pod was **not** in the list of pods to link. So it was never added to OTHER_LDFLAGS, and the native module isn't linked into the app at all (hence "native module not found" at runtime).

So the missing "Implicit dependency on target 'ExpoMifareScanner'" is the **same root cause** as the missing module in ExpoModulesProvider: autolinking (or the Podfile generator) did not include `@utapza/expo-mifare-scanner` when the Pods and xcconfig were generated. Fix that by:

1. Ensuring the **published** package has `expo-module.config.json` with `"apple": { "modules": ["ExpoMifareScannerModule"] }` and a valid podspec (e.g. `ios => '13.0'`, no Sentry dependency).
2. Reinstalling the app deps and running **pod install** again (or `npx expo prebuild --platform ios`) so the Podfile and `Pods-uTap.*.xcconfig` are regenerated. After that you should see `-l"ExpoMifareScanner"` in OTHER_LDFLAGS and an "Implicit dependency on target 'ExpoMifareScanner'" in the build log.

**Verify locally:** From the app root run `npx expo-modules-autolinking resolve --platform apple` and confirm `@utapza/expo-mifare-scanner` and pod `ExpoMifareScanner` appear. Then run `cd ios && pod install`. Then open `ios/Pods/Target Support Files/Pods-uTap/Pods-uTap.debug.xcconfig` and grep for `ExpoMifareScanner`; it should appear in OTHER_LDFLAGS and LIBRARY_SEARCH_PATHS.

## Why EAS might not autolink the module (and how to fix it)

On EAS, `node_modules` is populated from the **published** npm package (e.g. from the registry), not from your local `expo-mifare-scanner` repo. So the module is only autolinked on EAS if the **tarball** that EAS installs contains the right files and config.

1. **Published tarball is missing or wrong**
   - If the version on npm (e.g. `2.0.8` / `2.0.9`) was published **before** you added `"apple": { "modules": ["ExpoMifareScannerModule"] }` or fixed the podspec (e.g. `ios => '13.0'`), then EAS will get that old package and autolinking will either not find the module or will skip the pod (e.g. platform mismatch).
   - **Fix:** Publish a new version of `@utapza/expo-mifare-scanner` that includes the updated `expo-module.config.json` and `ios/ExpoMifareScanner.podspec`, then bump the app’s dependency to that version and trigger a new EAS build.

2. **`ios/` or podspec not in the tarball**
   - Autolinking looks for `*.podspec` inside top-level directories of the package (e.g. `ios/ExpoMifareScanner.podspec`). If `ios/` is not in the published package (e.g. omitted by `files` in `package.json` or by `.npmignore`), the module is found but **resolved as null** and not added to the Podfile.
   - **Fix:** Ensure `package.json` has `"files": [ ..., "ios" ]` and that `.npmignore` does not exclude `ios/`. Then republish.

### Verify what EAS actually installs

In the **expo-mifare-scanner** repo (the one you publish from):

```bash
npm pack --dry-run
```

Inspect the listed files; you should see at least:

- `expo-module.config.json` (with `"apple": { "modules": ["ExpoMifareScannerModule"] }` and `"ios": { "modules": ["ExpoMifareScannerModule"] }`)
- `ios/ExpoMifareScanner.podspec` (with `s.platforms = { :ios => '13.0' }` and no Sentry)
- `ios/ExpoMifareScanner/` (Swift sources)

Then pack for real and inspect the tarball:

```bash
npm pack
tar -tzf utapza-expo-mifare-scanner-*.tgz
```

If any of the above are missing, fix `package.json` / `.npmignore`, then publish a new version and use that version in the app before running another EAS build.

### Lockfile pins the version EAS installs

EAS (and local `npm ci`) installs the **exact** version in `package-lock.json`, not just the range in `package.json`. If the lockfile has:

```text
"resolved": "https://registry.npmjs.org/@utapza/expo-mifare-scanner/-/expo-mifare-scanner-2.0.8.tgz"
```

then EAS will always use **2.0.8** from npm. If 2.0.8 was published before you added `apple.modules` and the podspec fix, the iOS build will not autolink the module.

**Fix:** Publish a new version (e.g. 2.0.9) from the `expo-mifare-scanner` repo with the correct config and podspec, then in the app run:

```bash
npm install @utapza/expo-mifare-scanner@latest
```

Commit the updated `package.json` and `package-lock.json`, then run a new EAS build. The new tarball will be used and autolinking should include ExpoMifareScanner on iOS.
